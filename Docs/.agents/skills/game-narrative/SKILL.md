# Game Narrative



게임의 세계관 → 퀘스트 → 대사 → 분기를 한 번에 설계하는 메인 오케스트레이션 스킬이다.



## 목적



이 스킬은 다음과 같은 요청에 사용한다.



- 게임 시나리오 전체 설계

- 세계관부터 퀘스트, 대사, 엔딩까지 통합 설계

- 기존 세계관/스토리를 바탕으로 내러티브 요소 확장

- 스토리 문서 세트를 `_workspace/`에 정리



## 범위



포함:

- 세계관 설계

- 메인/사이드 퀘스트 설계

- NPC 대사 및 선택지 작성

- 분기 구조, 플래그, 엔딩 설계

- 내러티브 정합성 검토



제외:

- 게임 프로그래밍

- 레벨 디자인 및 맵 레이아웃

- 수치 밸런싱

- UI/UX 설계

- 전투 시스템 설계



## 관련 스킬



필요에 따라 아래 스킬을 함께 사용하거나 참조한다.



- `worldbuilding`

- `quest-design`

- `dialogue-writing`

- `branching-logic`

- `narrative-review`



## 입력 해석



작업 시작 시 사용자 요청에서 아래 정보를 최대한 추출한다.



- 게임 장르

- 세계관 방향

- 스토리 톤

- 목표 플랫폼 또는 표현 형식

- 예상 규모

- 기존 설정 또는 문서 존재 여부

- 산출물 전체 필요 여부 / 일부 단계만 필요 여부



불명확한 항목은 합리적으로 가정하되 문서에 명시한다.



## 산출물



결과물은 `_workspace/` 아래에 아래 순서로 저장한다.



1. `_workspace/00_input.md`

2. `_workspace/01_worldbuilding.md`

3. `_workspace/02_quest_design.md`

4. `_workspace/03_dialogue_script.md`

5. `_workspace/04_branch_map.md`

6. `_workspace/05_review_report.md`



## 기본 워크플로우



### Phase 1. 입력 정리

1. 사용자 요구를 구조화한다.

2. 기존 `_workspace/` 파일 존재 여부를 확인한다.

3. 이미 존재하는 파일이 유효하면 재사용하고, 필요한 단계만 새로 작성한다.

4. 입력 요약을 `_workspace/00_input.md`에 정리한다.



### Phase 2. 설계

1. 세계관 설계: `worldbuilding`

2. 퀘스트 설계: `quest-design`

3. 대사 작성: `dialogue-writing`

4. 분기 설계: `branching-logic`



대사와 분기 설계는 퀘스트 설계 완료 후 진행한다.



### Phase 3. 검증

1. 전체 산출물을 교차 검토한다.

2. `narrative-review` 기준으로 문제를 식별한다.

3. 필요 시 관련 문서를 수정한다.

4. 검토 결과를 `_workspace/05_review_report.md`에 기록한다.



## 작업 모드



### 전체 설계 모드

사용 예:

- "게임 시나리오 전체 설계해줘"

- "세계관, 퀘스트, 대사, 엔딩까지 만들어줘"



처리:

- 전체 워크플로우 수행



### 세계관 확장 모드

사용 예:

- "이 설정으로 세계관을 정리해줘"



처리:

- `00_input.md`

- `01_worldbuilding.md`

- 필요 시 `05_review_report.md`



### 퀘스트 설계 모드

사용 예:

- "이 세계관으로 메인 퀘스트를 짜줘"



처리:

- 기존 세계관 참조

- `02_quest_design.md`

- 필요 시 `05_review_report.md`



### 대사 작성 모드

사용 예:

- "이 퀘스트 기반으로 NPC 대사 써줘"



처리:

- 기존 세계관/퀘스트 참조

- `03_dialogue_script.md`

- 필요 시 `05_review_report.md`



### 분기 설계 모드

사용 예:

- "멀티 엔딩 분기 짜줘"

- "플래그 시스템 설계해줘"



처리:

- 기존 세계관/퀘스트 참조

- `04_branch_map.md`

- 필요 시 `05_review_report.md`



## 기존 파일 활용 규칙



- 사용자가 기존 설정이나 문서를 제공하면 우선 그것을 기준으로 작업한다.

- 기존 `_workspace/` 문서가 있으면 덮어쓰기보다 갱신을 우선한다.

- 이전 산출물이 새로운 입력과 충돌하면 충돌 지점을 명시한다.



## 품질 기준



- 세계관, 퀘스트, 대사, 분기가 서로 모순되지 않아야 한다.

- 각 단계의 산출물이 다음 단계의 입력으로 자연스럽게 이어져야 한다.

- 플레이어 관점에서 목표, 동기, 감정선, 결과가 이해 가능해야 한다.

- 최종 리뷰 보고서에는 최소한 주요 리스크와 수정 제안이 포함되어야 한다.



## 에러 핸들링



### 입력이 너무 모호한 경우

- 장르, 톤, 규모에 대해 가장 합리적인 기본안을 제시하고 그 기준으로 진행한다.

- 어떤 가정을 사용했는지 `00_input.md`에 적는다.



### 기존 IP 기반 요청인 경우

- 기존 설정을 우선 존중한다.

- 확장 가능한 부분만 설계한다.

- 원작과 충돌 가능성이 있는 지점은 명시한다.



### 산출물 일부만 가능한 경우

- 가능한 단계까지만 작성한다.

- 누락된 단계와 필요한 선행 정보를 명시한다.



## 출력 스타일



- 실무 문서처럼 구조화해서 작성한다.

- 서정적 문장보다 추적 가능한 문서 형식을 우선한다.

- 필요하면 표, 목록, ID 규칙을 사용한다.

- 문서 간 참조는 파일명 기준으로 일관되게 유지한다.
