# Branching Patterns



분기 설계에 사용하는 방법론 문서다.



## 왜 분기 로직이 중요한가



분기는 게임 내러티브를 다른 매체와 구분하는 핵심 요소다.  

하지만 분기가 많아질수록 제작량과 검증 비용이 급격히 늘어난다.  

적은 분기로 큰 차이를 체감하게 만드는 것이 핵심이다.



## 분기 구조 패턴 6가지



### 1. 병목 구조 (Bottleneck / Hub-and-Spoke)



```text

&#x20;   A

&#x20;  / 

&#x20; B   C

&#x20;   /

&#x20;   D

&#x20;  / 

&#x20; E   F

&#x20;   /

&#x20;   G

```



- 장점: 제작량을 통제하면서 선택의 자유를 준다.

- 핵심: 합류 후에도 이전 선택의 흔적이 남아야 한다.

- 적합: 장편 RPG, 구조가 큰 게임



### 2. 평행 경로 (Parallel Paths)



```text

A → B → C → D

A → E → F → D

A → G → H → D

```



- 장점: 리플레이 가치가 높다.

- 단점: 제작량이 커진다.

- 적합: 짧은 게임, 비주얼 노벨



### 3. 나뭇가지 구조 (Branching Tree)



```text

&#x20;     A

&#x20;    /|

&#x20;   B C D

&#x20;  /|   |

&#x20; E F   G H

```



- 장점: 직관적이다.

- 단점: 엔딩 수가 폭발하기 쉽다.

- 적합: 짧은 인터랙티브 픽션



### 4. 폴딩 구조 (Delayed Consequences)



```text

선택 시점 → 중간 진행 → 결과 발현

A에서 선택 → B, C, D → E에서 결과 반영

```



- 장점: 적은 분기로 큰 임팩트를 준다.

- 핵심: 플래그를 먼저 세우고 뒤늦게 반영한다.

- 적합: 장편 RPG, 에피소드형 게임



### 5. 상태 누적 구조 (State Accumulation)



```text

선택 1: 명예 +10 / 악명 +10

선택 2: 명예 +5  / 악명 +5

...

임계값 도달 시 엔딩 분기

```



- 장점: 단일 선택보다 행동 총합을 반영한다.

- 적합: 도덕성, 평판, 관계 시스템 중심 게임



### 6. 하이브리드 구조



실전에서는 여러 패턴을 혼합하는 경우가 많다.



- 메인 스토리: 병목 구조

- 서브 퀘스트: 짧은 나뭇가지

- 관계 시스템: 상태 누적

- 핵심 선택: 폴딩 구조



## 플래그 시스템 설계



### 플래그 네이밍 컨벤션



```text

[범주]_[대상]_[상태]



예시:

QUEST_bandit_camp_destroyed

REL_npc_aria_friendship_high

WORLD_bridge_repaired

PLAYER_has_ancient_sword

STORY_chapter2_twist_revealed

MORAL_saved_village

```



### 플래그 유형



| 유형 | 값 형태 | 사용처 | 예시 |

|------|--------|--------|------|

| 불리언 | true/false | 이벤트 발생 여부 | `QUEST_dragon_slain = true` |

| 카운터 | 정수 | 누적 행동 | `MORAL_mercy_count = 3` |

| 열거형 | 문자열 | 경로 선택 | `FACTION_chosen = "rebels"` |

| 타임스탬프 | 게임 시간 | 시간 제한 퀘스트 | `QUEST_deadline = day_30` |



### 플래그 의존성 매트릭스



```text

| 분기 지점 | 필요 플래그 | 조건 | 결과 |

|----------|-----------|------|------|

| CH3_결전 | REL_aria >= 50 AND QUEST_sword = true | 충족 | 아리아 합류 |

| CH3_결전 | REL_aria < 50 OR QUEST_sword = false | 미충족 | 단독 전투 |

```



## 엔딩 아키텍처



### 엔딩 분류



| 유형 | 달성 조건 | 목표 비율 |

|------|---------|---------|

| 정석 엔딩 | 메인 퀘스트 완료 | 다수 플레이어가 볼 수 있어야 함 |

| 최선 엔딩 | 정석 + 핵심 사이드 + 관계 최대 | 노력이 필요함 |

| 최악 엔딩 | 치명적 선택 누적 | 경고성/비극성 |

| 히든 엔딩 | 특수 조건 조합 | 낮은 발견율 |

| 진 엔딩 | 2회차 + 특수 조건 | 고난도 해금 |



### 엔딩 만족도 규칙



1. 모든 엔딩은 서사적으로 완결감이 있어야 한다.

2. 최선 엔딩은 노력 대비 보상이 있어야 한다.

3. 히든 엔딩은 최소한의 힌트를 가져야 한다.

4. 가능하면 엔딩 후 주요 선택 요약을 제공한다.



## 분기 테스트: 경로 시뮬레이션



```text

경로 1: A→B→D→F→최선엔딩

- 플래그: MORAL_good=5, REL_aria=80

- 누락 콘텐츠: 없음

- 플롯홀: 없음



경로 2: A→C→D→G→정석엔딩

- 플래그: MORAL_neutral=3, REL_aria=40

- 누락 콘텐츠: 일부 로맨스 서브 퀘스트

- 플롯홀: 특정 장면 조건 분기 필요

```
