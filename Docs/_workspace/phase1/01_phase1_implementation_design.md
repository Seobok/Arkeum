# 1단계 구현 설계안

## 문서 목적
- `07_development_roadmap.md`의 1단계 `코어 플레이 프로토타입`을 실제 Unity 구현 단위로 분해한다.
- 현재 프로젝트가 초기 템플릿 상태이므로, 첫 수직 프로토타입을 빠르게 검증할 수 있는 최소 아키텍처를 정의한다.
- 구현 전 확정이 필요한 모호 항목을 분리해 이후 재작업을 줄인다.

## 전제
- 엔진: Unity 6 (`6000.3.10f1`)
- 현재 코드 상태: 런타임 스크립트 없음, 샘플 씬만 존재
- 목표: `10분 이상 반복 플레이 가능한 회귀 루프` 검증
- 범위: 거점 1개, 던전 테스트 구역 1~2개, 적 2~3종, 무기 타입 1종, 소모품 2종(`응고 지혈포`, 던전 상인용 회복 소모품)

## 1단계 구현 목표 재정의
- 플레이어는 거점에서 런을 시작한다.
- 던전에서 `행동 1회 = 시간 1단위` 규칙으로 이동, 공격, 대기, 상호작용을 수행한다.
- 적은 플레이어 행동 후 같은 규칙으로 반응한다.
- 런 도중 `혈편`을 얻고, 던전 상인에게 사용한다.
- 사망 또는 강제 종료 조건 발생 시 런이 종료되고 `귀환 제단`으로 복귀한다.
- 런 결과를 정산해 `잔광`만 영구 보존한다.
- 거점에서 `잔광`으로 `응고 지혈포` 시작 해금을 수행하고 즉시 재도전할 수 있다.

## 구현 전략

### 1. 수직 슬라이스 대신 규칙 프로토타입 우선
- 1단계는 콘텐츠 완성보다 `규칙이 끊기지 않고 이어지는지`가 중요하다.
- 따라서 아트/연출보다 아래 요소를 먼저 고정한다.
  - 턴 해석
  - 런 상태 전이
  - 자원 정산
  - 저장/복원
  - 최소 UI 피드백

### 2. 데이터 중심 구조
- 1단계부터 전투 수치와 아이템 효과를 코드 하드코딩으로만 묶으면 2단계 이후 퀘스트/분기 확장 때 비용이 커진다.
- 런타임 로직은 코드로 두되, 적/아이템/무기/던전 구역은 `ScriptableObject` 데이터로 분리한다.

### 3. 씬 분리보다 단일 플레이 씬 + 상태 전환
- 첫 빌드에서는 씬을 과하게 쪼개지 않는다.
- 권장 구성:
  - `Bootstrap`
  - `Hub`
  - `Dungeon`
- `Bootstrap`은 저장 로드와 초기화만 담당한다.
- `Hub`와 `Dungeon`은 명확히 분리해 회귀 흐름과 상태 복귀를 단순화한다.

## 핵심 시스템 설계

### A. 게임 상태와 런 상태
- 최상위 상태 머신:
  - `Boot`
  - `Hub`
  - `RunPreparing`
  - `InRun`
  - `RunResult`
- 런 상태 전이:
  1. 거점 진입
  2. 해금/준비 반영
  3. 던전 로드
  4. 행동 기반 진행
  5. 사망 또는 강제 종료 조건 발생
  6. 정산
  7. 거점 복귀

### B. 행동 기반 진행 시스템
- 1단계에서는 엄격한 `플레이어 행동 후 월드 갱신` 구조로 구현한다.
- 한 행동 단위에 포함되는 것:
  - 이동 1칸 또는 1노드
  - 기본 공격 1회
  - 아이템 사용 1회
  - 상호작용 1회
  - 대기 1회
- 행동 처리 순서:
  1. 입력 검증
  2. 행동 수행
  3. 플레이어발 즉시 효과 적용
  4. 적 AI 갱신
  5. 환경/상태 이상 갱신
  6. 사망/전투 종료 판정
  7. UI 갱신

### C. 전투 규칙
- 1단계는 단순 근접 전투 1종으로 제한한다.
- 필요한 최소 수치:
  - 체력
  - 공격력
  - 방어 또는 피해 경감
  - 사거리
  - 행동 우선권은 고정
- 권장 규칙:
  - 명중/회피 확률 없이 확정 피해 기반
  - 플레이어와 적 모두 행동 1회당 공격 1회
  - 피격 반응은 연출보다 상태 갱신 우선
- 이유:
  - 리듬 검증이 목적이므로 랜덤 편차를 최소화해야 피드백이 선명해진다.

### D. 적 AI
- 1단계 적 유형 권장:
  - 근접 추적형
  - 느리지만 강한 경로 차단형
  - 상인/비전투 NPC
- 공통 AI 흐름:
  1. 플레이어 감지 여부 확인
  2. 인접 시 공격
  3. 비인접 시 추적
  4. 감지 실패 시 대기 또는 순찰
- 경로탐색은 첫 빌드에서 단순화한다.
  - 격자형이면 Manhattan 거리 기반 한 칸 추적
  - 자유 이동형이면 NavMesh 대신 노드 기반 이동 권장

### E. 자원 시스템

#### `혈편`
- 런 전용 자원
- 획득처:
  - 적 처치
  - 테스트 오브젝트 파괴
- 사용처:
  - 던전 상인에게 회복 소모품 구매
- 소멸:
  - 사망
  - 런 포기

#### `잔광`
- 영구 자원
- 획득처:
  - 런 종료 정산
  - 최고 도달 구역 보너스
- 사용처:
  - `응고 지혈포` 시작 해금 1종
- 저장:
  - 프로필 저장에 즉시 반영

### F. 거점 시스템
- 1단계 거점 기능은 메뉴 허브에 가깝게 구현한다.
- 필수 화면:
  - 현재 보유 `잔광`
  - 시작 해금 상태
  - 직전 런 결과
  - 장례자 대화
  - 출발 버튼
- 거점 상태 변화는 최소 수준으로 시작한다.
  - 회귀 횟수에 따른 장례자 대사 2~3변주
  - 해금 전/후 제단 문구 변화

### G. 저장 시스템
- 저장 파일은 최소 2개로 분리한다.
  - `ProfileSave`
  - `RunSave`
- `ProfileSave`
  - 총 회귀 횟수
  - 최고 도달 구역
  - 잔광 총량
  - 시작 해금 여부
  - 최소 퀘스트/플래그
- `RunSave`
  - 현재 씬/구역
  - 플레이어 체력
  - 런 내 자원(`혈편`)
  - 장비/소모품
  - 현재 턴 카운트
- 1단계에서는 `거점 영구 저장`만 우선 구현한다.
- 추후 확장 시 `새 층 진입 시 자동 저장`을 추가하는 방향으로 설계한다.
- 그 외 수동 저장과 런 도중 임의 저장은 지원하지 않는다.

### H. UI/UX
- 1단계 UI 목표는 화려함이 아니라 규칙 이해다.
- 표현 방식은 `HUD 중심`으로 확정한다.
- 필수 UI:
  - HP
  - 런 내 `혈편`
  - 영구 `잔광`은 거점에서만 노출
  - 행동 안내 문구: `행동 후 적이 반응합니다`
  - 런 종료 결과창: 잃은 것 / 남는 것 분리
  - 초반 목표 문구: `잿빛 회랑으로 내려가 네 이름에 반응한 잔광을 회수하라.`
- 결과창은 반드시 두 영역으로 분리한다.
  - 이번 런에서 사라진 것
  - 회귀 후 남은 것

## 리스크와 대응

### 리스크 1. 행동 기반 진행이 답답하게 느껴질 수 있음
- 대응:
  - 입력 후 반응 시간을 짧게 유지
  - 적 반응을 한 프레임에 일괄 처리
  - 대기 행동과 이동 체감을 명확히 구분

### 리스크 2. 장비 소실이 손해만 크게 느껴질 수 있음
- 대응:
  - 결과창에서 `남은 것`을 먼저 보여준다.
  - 첫 `잔광` 해금 보상을 빠르게 체감시킨다.

### 리스크 3. 1단계에서 시스템 수가 과도하게 늘어날 수 있음
- 대응:
  - 퀘스트는 `MQ-01` 문구 노출 수준으로 제한
  - 캐릭터 해금, 세력 분기, 중층 데이터는 인터페이스만 남기고 미구현 유지

## 확정 반영 사항

### 1. 이동 규칙
- 이동은 `격자 기반 1칸 이동`으로 확정한다.

### 2. 전투 거리 규칙
- 첫 빌드 전투는 `근접 무기 1종`만 구현한다.

### 3. 런 종료 조건
- `자발 귀환`은 구현하지 않는다.

### 4. 던전 구조
- 장기적으로는 `절차 생성`이 필요하다.
- 1단계는 반드시 `수제 테스트 구역`으로 고정한다.

### 5. 전투 수치 철학
- 1단계는 명중, 치명타, 회피 같은 확률 규칙 없이 `확정 피해 기반`으로 구현한다.

### 6. 저장 정책
- 1단계는 `거점 영구 저장`만 구현한다.
- 추후 `새 층 진입 시 자동 저장`으로 확장한다.

### 7. UI 표현 수준
- 1단계 UI는 `HUD 중심`으로 확정한다.

## 권장 확정안
- 이동: 격자 기반
- 전투: 근접 무기 1종
- 런 종료: 자발 귀환 없음
- 던전: 1단계는 수제 테스트 구역 2개, 추후 절차 생성 확장
- 전투 수치: 확률 없는 확정 피해 기반
- 저장: 거점 영구 저장 우선, 추후 층 진입 자동 저장
- UI: HUD 중심

위 항목이 확정되면 1단계 구현을 바로 시작할 수 있다.
