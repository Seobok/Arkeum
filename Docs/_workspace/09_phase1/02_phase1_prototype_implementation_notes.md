# 1단계 프로토타입 구현 정리

## 문서 목적
- 1단계 프로토타입을 실제로 어떤 단위로 구현했는지 기록한다.
- 각 구현 방식에 대해 대안 구조와 장단점을 비교하고, 이번 프로토타입에서 왜 현재 구조를 선택했는지 남긴다.
- 이후 2단계 이상에서 교체하거나 확장해야 할 지점을 미리 구분한다.

## 구현 결과 개요
- 현재 프로토타입은 `SampleScene`에 직접 오브젝트를 배치하는 방식이 아니라, 플레이 시작 시 코드를 통해 테스트 월드와 HUD를 생성하는 방식으로 구현했다.
- 지원 범위:
  - 거점 상태
  - 런 시작
  - 격자 기반 1칸 이동
  - 행동 1회당 적 반응
  - 근접 공격
  - 적 2종 AI
  - `혈편` 획득 및 던전 상인 사용
  - `잔광` 정산 및 시작 소모품 해금
  - 런 종료 결과 화면
  - 거점 영구 저장

## 구현 단위별 설명

### 1. 부트스트랩
- 구현:
  - `PrototypeBootstrap`이 씬 로드 후 `PrototypeGameController`를 자동 생성한다.
- 선택 이유:
  - 현재 프로젝트는 실제 런타임 씬 셋업이 전혀 없는 상태였다.
  - Unity 에디터에서 수동으로 씬, 프리팹, UI를 배치하는 작업 없이 바로 프로토타입을 실행 가능한 형태로 만드는 것이 우선이었다.
- 대안 1. 씬 수동 배치 기반
  - 장점:
    - Unity 사용 방식에 더 정석적이다.
    - 디자이너가 Inspector에서 구조를 바로 확인하기 쉽다.
  - 단점:
    - 현재 작업 환경에서는 에디터 조작 결과를 재현하거나 검증하기 어렵다.
    - 씬 YAML 편집 비용이 높고, 사소한 수정도 충돌이 커진다.
- 대안 2. Addressables/서비스 로더 기반
  - 장점:
    - 장기 구조 확장성은 좋다.
  - 단점:
    - 1단계 목표 대비 지나치게 무겁다.
- 결론:
  - 1단계는 `코드 부트스트랩`이 가장 빠르고 안전하다.

### 2. 게임 상태 관리
- 구현:
  - `GamePhase`를 `Hub`, `InRun`, `RunResult` 세 단계로 단순화했다.
  - 상태 전환은 `PrototypeGameController`가 직접 관리한다.
- 선택 이유:
  - 이번 단계의 핵심은 `거점 -> 런 -> 결과 -> 거점` 순환 검증이다.
  - 여기서는 범용 상태 머신 프레임워크보다 상태 수를 강하게 제한하는 편이 디버깅이 쉽다.
- 대안 1. 일반화된 상태 머신 클래스 도입
  - 장점:
    - 추후 상태가 늘어나도 구조가 깔끔하다.
  - 단점:
    - 현재 상태 수가 적어 오히려 추상화 비용이 더 크다.
- 대안 2. 씬 전환 중심 상태 관리
  - 장점:
    - 거점/던전 분리가 명확하다.
  - 단점:
    - 에디터 셋업과 씬 자산 의존도가 높아진다.
- 결론:
  - 1단계에서는 `단일 컨트롤러 내부 상태 전환`이 가장 실용적이다.

### 3. 월드 구성
- 구현:
  - `DungeonLayout`이 걷기 가능한 격자와 구역 깊이 정보를 가진다.
  - 던전은 수제 테스트 구역 2개를 코드로 생성한다.
- 선택 이유:
  - 문서에서 확정된 `격자 기반 1칸 이동`과 가장 잘 맞는다.
  - AI, 충돌, 상호작용 검증이 단순해진다.
- 대안 1. 자유 이동 + 턴 소비
  - 장점:
    - 액션 감각을 더 빨리 체험할 수 있다.
  - 단점:
    - 행동 단위 턴 규칙의 명확성이 약해지고, 적 반응 타이밍이 모호해진다.
- 대안 2. 절차 생성 맵
  - 장점:
    - 반복 플레이 감각을 빠르게 볼 수 있다.
  - 단점:
    - 1단계에서 검증해야 할 것은 맵 다양성이 아니라 코어 루프의 안정성이다.
    - 디버깅 난도가 급격히 올라간다.
- 결론:
  - 현재 단계는 `수제 격자 맵`이 맞다.
  - 절차 생성은 2단계 이후 별도 시스템으로 분리하는 편이 낫다.

### 4. 턴 처리
- 구현:
  - 플레이어의 이동, 공격, 대기, 아이템 사용, 상호작용을 모두 1행동으로 처리했다.
  - 행동 후 즉시 적 AI를 일괄 갱신한다.
- 선택 이유:
  - 문서의 `행동이 곧 시간` 규칙을 가장 직접적으로 표현하는 방식이다.
  - 플레이어가 매 입력의 비용을 명확히 체감할 수 있다.
- 대안 1. 에너지/속도 수치 기반 턴 시스템
  - 장점:
    - 빠른 적, 느린 적 같은 차이를 더 정교하게 표현 가능하다.
  - 단점:
    - 현재 프로토타입 단계에서는 계산이 보이지 않아 오히려 이해가 어려워진다.
- 대안 2. 실시간 + 입력 후 일시정지 하이브리드
  - 장점:
    - 연출 감각이 좋다.
  - 단점:
    - 1단계 문서 목표와 충돌한다.
- 결론:
  - `플레이어 1행동 후 월드 전체 반응`이 가장 적합하다.

### 5. 전투
- 구현:
  - 근접 공격 1종만 구현했다.
  - 명중, 회피, 치명타 없이 확정 피해만 적용한다.
  - 피해량은 `공격력 - 방어력` 하한 1 방식이다.
- 선택 이유:
  - 1단계의 검증 포인트는 전투 리듬과 회귀 감각이지, 전투 확률 설계가 아니다.
  - 확률을 제거해야 피드백이 선명하다.
- 대안 1. 명중/회피/치명타 포함
  - 장점:
    - 전투 감정 기복이 빨리 생긴다.
  - 단점:
    - 패배 원인이 시스템 구조인지 운인지 구분하기 어려워진다.
- 대안 2. 스킬 기반 전투
  - 장점:
    - 캐릭터 특성을 빨리 보여줄 수 있다.
  - 단점:
    - 1단계 범위를 벗어난다.
- 결론:
  - `확정 피해 기반 근접 전투`가 가장 타당하다.

### 6. 적 AI
- 구현:
  - 적은 `추적형`과 `중장 추적형` 두 종류다.
  - 인접 시 공격, 비인접 시 플레이어 쪽으로 1칸 전진한다.
  - 중장형은 `ActionInterval = 2`로 느리지만 강하게 동작한다.
- 선택 이유:
  - 적 종류 차이를 수치와 속도로 바로 체감할 수 있다.
  - 경로탐색을 단순화하면서도 플레이 리듬 차이는 유지된다.
- 대안 1. A* 기반 정교한 경로탐색
  - 장점:
    - 복잡한 맵에서도 자연스럽다.
  - 단점:
    - 현재 맵 구조에서는 과하다.
- 대안 2. 상태 트리 기반 AI
  - 장점:
    - 이후 확장성이 좋다.
  - 단점:
    - 1단계 검증 범위를 크게 초과한다.
- 결론:
  - 현재는 `한 칸 추적 AI`가 맞다.

### 7. 자원과 정산
- 구현:
  - `혈편`은 적 처치로 얻고, 던전 상인에게서 회복 소모품 구매에만 사용한다.
  - `잔광`은 런 종료 시 정산으로만 얻고, 거점 해금에 사용한다.
  - 런 결과창에서 `사라진 것`과 `남은 것`을 명시적으로 분리했다.
- 선택 이유:
  - 문서의 가장 중요한 UX 요구 중 하나가 `런 내 자원`과 `영구 자원`의 구분이다.
  - 시스템보다 결과 화면이 이 구분을 학습시키는 데 더 중요하다.
- 대안 1. 하나의 통합 재화만 사용
  - 장점:
    - 구현은 쉽다.
  - 단점:
    - 이 게임의 핵심 감각인 `소실과 보존의 이중 구조`가 사라진다.
- 대안 2. 상점/성장/제작을 동시에 구현
  - 장점:
    - 메타 시스템이 빨리 풍성해진다.
  - 단점:
    - 1단계에서는 무엇이 재미인지 판단이 흐려진다.
- 결론:
  - `혈편/잔광 이원화`는 반드시 유지하는 편이 맞다.

### 8. 저장
- 구현:
  - `PrototypeSaveService`가 `ProfileSaveData`를 JSON으로 저장한다.
  - 저장 시점은 거점 해금 구매와 런 종료 시점만 사용한다.
  - `RunSnapshot` 타입은 남겨두되, 현재는 런 중 저장에 사용하지 않는다.
- 선택 이유:
  - 문서 확정사항이 `거점 영구 저장 우선`이기 때문이다.
  - 이후 `새 층 진입 시 자동 저장`으로 확장할 여지를 남기기 위해 데이터 타입은 미리 분리했다.
- 대안 1. 런/프로필 통합 저장
  - 장점:
    - 파일 수가 적다.
  - 단점:
    - 이후 계층 저장 확장 시 구조가 지저분해진다.
- 대안 2. 런 저장까지 즉시 구현
  - 장점:
    - 이어하기가 빨리 된다.
  - 단점:
    - 현재 1단계 목표와 무관하고, 디버깅 포인트만 늘어난다.
- 결론:
  - `프로필 우선 저장 + 런 저장 타입만 예약`이 적절하다.

### 9. UI
- 구현:
  - HUD와 결과창은 `OnGUI` 기반 즉시 모드 UI로 만들었다.
- 선택 이유:
  - 현재 작업 환경에서 프리팹, 캔버스, 앵커를 에디터 없이 안정적으로 구성하는 것보다 코드 생성 UI가 훨씬 빠르다.
  - HUD 중심이라는 문서 기준은 유지하면서, 씬 자산 의존도를 줄일 수 있다.
- 대안 1. uGUI 캔버스 코드 생성
  - 장점:
    - Unity 표준 UI 구조와 가깝다.
  - 단점:
    - 텍스트, 레이아웃, 리사이즈까지 모두 코드로 만들면 오히려 더 장황하다.
- 대안 2. UI Toolkit
  - 장점:
    - 구조적이고 스타일 분리가 쉽다.
  - 단점:
    - 1단계 프로토타입 규모에서는 초기 셋업 비용이 더 크다.
- 결론:
  - 이 단계는 `OnGUI`가 가장 빠른 검증 도구다.
  - 2단계부터는 uGUI나 UI Toolkit으로 옮기는 것이 맞다.

## 현재 구조의 장점
- 에디터 없이도 바로 플레이 가능한 최소 프로토타입이다.
- 문서에서 확정한 규칙이 코드에서 한눈에 대응된다.
- 씬/프리팹 자산 충돌이 적다.
- 이후 실제 제작용 구조로 갈아탈 때 어느 부분이 임시인지 명확하다.

## 현재 구조의 한계
- `PrototypeGameController`에 책임이 많이 모여 있다.
- HUD가 즉시 모드 UI라 최종 구조로 가져가기 어렵다.
- 적 데이터와 아이템 데이터가 코드에 고정돼 있다.
- 단일 맵, 단일 캐릭터, 단일 무기 전제에 강하게 묶여 있다.

## 2단계 이상에서 분리해야 할 부분
- `PrototypeGameController`를 상태/전투/던전/상호작용 서비스로 분리
- 적/아이템/무기 데이터를 `ScriptableObject`로 이동
- 거점과 던전을 씬 단위 또는 서브씬 단위로 분리
- HUD를 정식 UI 시스템으로 이관
- 새 층 진입 시 `RunSave` 자동 저장 활성화
- 퀘스트 및 플래그 시스템을 범용 구조로 확장

## 이번 구조를 유지한 이유 요약
- 1단계 목표는 `코어 플레이 프로토타입` 검증이다.
- 지금 필요한 것은 확장성이 아니라 `짧은 시간 안에 플레이 감각을 실제로 확인할 수 있는 최소 구조`다.
- 따라서 장기적으로 더 좋은 구조가 있더라도, 이번 단계에서는 `코드 부트스트랩 + 수제 격자 맵 + 확정 피해 전투 + 즉시 모드 HUD` 조합이 가장 효율적이었다.
